> Applies the universal 9-step process in `README.md` to Tips & Tricks briefs. This file owns the structural template only.

## Structural template

**Hook → Problem/Context → The Tip (step-by-step) → Quick Proof/Demo → CTA**

| Section | Job | Notes |
|---|---|---|
| **Hook** | Land the brief's `hook_direction` immediately | Usually Direct Value Promise (`03_IdeaEngine/Hook_System.md`) — Tips content earns attention by promising a specific, fast win, so the hook should name the win, not tease it vaguely |
| **Problem/Context** | The frustration or gap this tip resolves, in one or two lines | Kept tight — Tips is the shortest-form category; this section orients, it doesn't build a full case the way Tool Review's Problem section does |
| **The Tip** | The actual technique, step by step | The core content; must be specific and actionable, not generic advice — see `02_Research/Workflows.md`'s Tips workflow on verifying the tip is still current before it reaches this stage |
| **Quick Proof/Demo** | Show the tip working | Brief — enough to confirm it's real, not a full walkthrough |
| **CTA** | Single ask | Typically: try it now, or a related tip/video if one exists |

## Category-specific notes

- **Staleness is the primary risk in this category.** `02_Research/Workflows.md` flags Tips as having the highest staleness risk of any category — if, during Extract Facts (README.md step 2), anything about the tip's current validity looks uncertain, that's a Failure Recovery case, not something to soften with hedging language.
- **Caveats stay in, even in a short format.** If a tip only works under certain conditions, that caveat is stated plainly (per `README.md` § Examples/Edge Cases) rather than cut for brevity — a tip that's misleading because a caveat was trimmed is worse than a slightly longer, accurate one.
- **Multiple independent tips in one brief:** per `02_Research/Workflows.md`, a single Tips fact-list often yields several small, unrelated quick wins. If `03_IdeaEngine` bundles more than one into a single brief, Script Engine should treat each as its own Hook→Problem→Tip→Proof cycle within the video rather than blending them into one continuous narrative — keeps each tip independently useful even if a viewer only watches part of the video.
