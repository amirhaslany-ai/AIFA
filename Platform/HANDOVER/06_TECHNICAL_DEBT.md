# 06 — Technical Debt

Every item below was verified against the current source, not assumed from prose. Ranked by how much harm it would cause if the system were deployed today without addressing it first.

## Critical

1. **The entire persistence layer has never been exercised against a real database.** No Postgres instance has ever been available in any environment this codebase was developed in. All 5 migrations were generated via `prisma migrate diff` (schema-to-schema comparison, no live DB required) and never applied (`prisma migrate deploy`). Every repository (`Prisma*Repository`) is verified only by reading its generated SQL and by unit tests against in-memory fakes that mirror its intended behavior — never a real query. **Before any real deployment, every repository must be exercised against a real Postgres instance at least once**, including concurrent-write scenarios (the ledger idempotency logic, the message/usage-event unique constraints) that an in-memory fake can approximate but not prove.
2. **No rate limiting exists anywhere.** `/v1/auth/login`, `/v1/auth/register`, and `/v1/chat` all currently accept unlimited requests from any caller. Deployed as-is, this is a brute-force login vector and an unbounded wallet-draining/cost vector (a caller can spam `POST /v1/chat` as fast as the network allows, each call triggering a real provider request once a real vendor key is configured).
3. **CI has never executed, not even once.** The workflow file is correct and correctly located, but the local `main` branch is 8 commits ahead of `origin/main` (verified: `git rev-list --left-right --count origin/main...main` → `0 8`). Every "lint/typecheck/test/build passes" claim in this entire repository, including this handover package, is based on running the equivalent commands locally — GitHub Actions has never scanned this workflow.

## High

4. **Zero integration or end-to-end tests.** Every test in the repository (~104 total) is either a pure unit test or an "integration-style" test against an in-memory port fake. No test has ever run against a real Postgres, a real Redis, or a real HTTP server. `docs/architecture/testing-architecture.md` documents a target Testcontainers strategy that has never been built.
5. **Docker has never been built.** Both Dockerfiles contain real, specific fixes for bugs found by inspection (missing `tsconfig.base.json` and `prisma/schema.prisma` in the pruned build context) — but the actual `docker build`/`docker run` cycle has never executed, because no Docker daemon has been available in any development environment used. Treat both images as unverified until built and run for the first time.
6. **Ephemeral JWT signing keys if env vars are unset.** If `AUTH_JWT_PRIVATE_KEY_PEM`/`AUTH_JWT_PUBLIC_KEY_PEM` are not set, `JwtKeyProvider` silently generates a new Ed25519 keypair every process start and only logs a warning — it does not fail to start. In a real deployment, forgetting to set these means every token becomes invalid on every restart/redeploy, silently logging users out, with no hard failure to force the operator to notice before going live.
7. **A chat account can go negative from a single expensive call.** By design (ADR-0014), Chat only checks that the balance is strictly positive before calling a provider, then debits the real cost afterward with no upper bound. There is currently no per-account spending cap, no maximum-cost-per-call guard, and no alerting if a wallet goes deeply negative. This is a documented, deliberate simplification, not an oversight — but it is real financial exposure that must be a founder decision before real money is at stake (see `12_OPEN_DECISIONS.md`).
8. **`AiProviderConfig` changes require an application restart to take effect**, contradicting `docs/architecture/ai-provider-layer.md`'s explicit claim that "disabling a failing provider or reordering fallback priority is an operational action, not a deploy." `ProviderRegistryAdapter.onModuleInit()` reads the table exactly once, at boot, with no refresh mechanism. An operator disabling a misbehaving provider today must restart the process, not just update a database row.
9. **No authorization model beyond "you own the resource."** There is no role/permission concept anywhere in the codebase. This is fine for a single-tier consumer product but must be resolved before any admin tooling, team/org accounts, or tiered-access feature is built — retrofitting authorization onto an authorization-less system is materially harder than building it in from the start.

## Medium

10. **The dependency (database/Redis) health check has no caching**, unlike the provider health check which has a 5-second cache. `DependencyHealthAdapter.checkAll()` performs a live `SELECT 1` and a live Redis `ping()` on every single call to `/v1/health/ready`. An orchestrator probing readiness every few seconds across multiple replicas will generate continuous live database/Redis round-trips purely for health checks. This is the same class of problem the pre-Sprint-1 audit flagged for the provider side (and which was fixed there) — it was not carried over to the dependency-health side when that check was built.
11. **No security headers.** No `helmet` (or equivalent) is installed or configured — no `Content-Security-Policy`, `X-Frame-Options`, `Strict-Transport-Security`, etc. on any response.
12. **Swagger UI (`/v1/docs`) is unauthenticated** and would expose the entire API surface (including internal error-shape details) to anyone who can reach the server, in any environment including a hypothetical production deployment. No environment-based gating exists to disable or protect it outside development.
13. **No secret-scanning in CI.** Carried over unresolved from the pre-Sprint-1 audit (`TECHNICAL_DEBT.md` item 5) — still true today. No `gitleaks` or equivalent step exists in `.github/workflows/aifa-platform-ci.yml`.
14. **Logging is not wired into Nest's DI container.** Only the global exception filter (`DomainErrorFilter`) receives a `Logger` instance. Successful requests are never logged with their correlation id (`requestId`) — the request-id middleware exists and generates/propagates the id, but nothing logs it on the happy path, so you can trace an error back to a request but not trace a request's full lifecycle.
15. **Prisma is never explicitly disconnected.** No `app.enableShutdownHooks()` call in `main.ts`, no `OnModuleDestroy` handler for the Prisma client (contrast with `RedisHealthIndicator`, which does implement `OnModuleDestroy` correctly). A rolling deploy or container stop does not drain the database connection cleanly.
16. **Neither Dockerfile runs as a non-root user.** No `USER` directive exists in either `apps/api/Dockerfile` or `apps/web/Dockerfile` — both containers, if built and run, would run their process as root.
17. **No dependency-vulnerability scanning.** No Dependabot or Renovate configuration exists anywhere in the repository.
18. **The root-level pre-Sprint-1 audit documents are stale and now factually wrong about current implementation state**, yet remain present at the repository root alongside the accurate, current documentation. `01_FINAL_ARCHITECTURE_AUDIT.md` through `05_ARCHITECTURE_FREEZE_DECISION.md`, `ARCHITECTURE_REVIEW.md`, `TECHNICAL_DEBT.md` (the old one, not this file), `IMPROVEMENT_PLAN.md`, and `PROGRESS_REPORT.md` all describe a codebase with "no Wallet/Pricing/Auth/AI-provider code" — all now false. A reader who finds these before finding `HANDOVER/` will draw wrong conclusions. Recommend either deleting them, or adding a one-line banner to each pointing at this package, once this handover has been reviewed and accepted.

## Low

19. **Duplicated `.env`-loading logic.** `apps/api/src/main.ts` and `apps/web/next.config.mjs` each hand-roll loading the monorepo-root `.env` via `dotenv` + a relative `__dirname` path, instead of sharing one `loadRootEnv()` helper in `@aifa/config`. Carried over unresolved from the pre-Sprint-1 audit.
20. **`apps/web` declares `@aifa/config` and `@aifa/logger` as dependencies but never uses either in actual route code** (only `next.config.mjs`'s dotenv loading touches the env at all, directly via `process.env`, not through `@aifa/config`). Not urgent since `apps/web` has no real product code yet, but the declared-but-unused coupling should be resolved (either use them or drop them) once real frontend work begins.
21. **`CircuitBreaker`'s half-open state admits unlimited concurrent trial requests**, not the single probe a classic half-open gate implies. A flapping provider whose cooldown just expired gets hit by every concurrent request in that window, not one. Carried over unresolved from the pre-Sprint-1 audit.
22. **`docs/architecture/package-boundaries.md`'s rule that only `apps/api/src/infrastructure/persistence/*` (and `infrastructure/providers/*` for `AiProviderConfig`) may import `@aifa/database` is documented but not machine-enforced by ESLint.** Unlike the hexagonal layer rule and the vendor-SDK rule, nothing currently stops any file anywhere from importing `@aifa/database` directly.
23. **ADR-0012's expand/contract migration discipline has never actually been exercised.** All 5 real migrations to date are purely additive. The policy is sound but unproven — a genuinely destructive/rename migration has never been attempted under it.
24. **Pricing has no campaign/discount rules or per-plan multipliers** — deliberately not invented (no real business requirements exist), but flagged here so it isn't mistaken for a forgotten feature versus an explicit scope cut awaiting a founder decision.
25. **Chat has no full request-level idempotent replay** — a retried `messageId` is rejected (409), not replayed with the original response. A deliberate, documented scope cut (ADR-0014), not a bug, but worth knowing before assuming a client can safely retry-and-forget a `POST /v1/chat` call.
26. A stray, empty, untracked directory (`Platform/webtest`) exists on disk. Not part of git history, contains nothing. Harmless; delete at will.
