# 03 — Implementation Status

Status legend: **IMPLEMENTED** (real code, real tests, wired end-to-end within what this environment could verify) / **PARTIALLY IMPLEMENTED** (some real code exists but a meaningful part is missing, stubbed, or unverified) / **PLANNED** (a design document exists but zero code) / **NOT IMPLEMENTED** (neither code nor a design document exists, or the concept is out of scope for this repository entirely).

| Subsystem | Status | Detail |
|---|---|---|
| **Authentication** | IMPLEMENTED | Real argon2id password hashing (`@node-rs/argon2`), real Ed25519 JWT signing/verification (`jose`), refresh-token rotation with reuse detection (family-based revocation). Endpoints: `POST /v1/auth/register`, `/login`, `/refresh`, `/logout`, `GET /v1/auth/me`. **Caveat:** if `AUTH_JWT_PRIVATE_KEY_PEM`/`AUTH_JWT_PUBLIC_KEY_PEM` are unset, an ephemeral keypair is generated per process — tokens don't survive a restart. This is logged as a warning but is a real production footgun if someone forgets to set real keys. Never live-verified against a real Postgres (see `10_PRODUCTION_READINESS.md`). |
| **Authorization** | PARTIALLY IMPLEMENTED | Authentication (who are you) is real. Authorization beyond "you may only read/write your own `accountId`-scoped resources" **does not exist** — no roles, no permissions, no admin concept, no `PermissionPort` (mentioned in `api-standards.md`'s design but never built). Every endpoint's access model is "authenticated + owns the resource," nothing finer-grained. |
| **Wallet** | IMPLEMENTED | Ledger-based `credit`/`reserve`/`settle`/`rollback`/`debit`, real Prisma-transaction persistence, idempotency via `(walletId, referenceId, type)` uniqueness. Only `GET /v1/wallet` is public; the mutation use cases are real but deliberately not exposed over HTTP (see `02_ARCHITECTURE_MAP.md`). No real payment ever moved money — see **Payment** below. |
| **Ledger** | IMPLEMENTED | Append-only `LedgerEntry`, five entry types, `Wallet.balanceMinorUnits` is a reconcilable cache. Real regression tests cover retry-safety (no double-credit/double-debit on a repeated call). |
| **Pricing** | IMPLEMENTED (partial scope) | Ordered rule pipeline: `BaseMarkupRule` → `FloorRule`, config-driven multiplier/floor, basis-points integer math with ceiling rounding. **No campaign/discount rules, no per-plan multipliers exist** — deliberately not invented (no fake business data). No HTTP surface; invoked only by `SendChatMessageUseCase`. |
| **Payment** | NOT IMPLEMENTED | No payment gateway integration (Stripe, etc.), no webhook handler, no real money has ever moved. `CreditWalletUseCase` is the only way funds enter a wallet, and it has no HTTP endpoint — funding a wallet today requires calling the use case directly from code (e.g. a test), not something an end user or even an operator can currently trigger via the running system. |
| **AI Providers** | PARTIALLY IMPLEMENTED | One real HTTP adapter (`OpenAiCompatibleAdapter`, `fetch`-based, OpenAI-compatible Chat Completions wire format), selected per `AiProviderConfig` row only when fully configured with a live env-var-referenced API key; otherwise falls back to `StubAdapter`. **No real vendor API key has ever been used against it in this codebase's history** — verified only via unit tests against an injected-`fetch` double. No streaming, no capability matrix, no retry policy, no response cache — all explicitly design-only (ADR-0007). |
| **Provider Registry** | IMPLEMENTED | Config-driven provider bootstrap, `CircuitBreaker` (3-failure threshold, 30s cooldown) wrapping every registered provider, `FallbackChain` trying providers in priority order. **Config is read once at boot only** — see the note in `02_ARCHITECTURE_MAP.md`; a live provider-config change requires a restart, contradicting the architecture doc's "operational action, not a deploy" claim. |
| **Conversation** | IMPLEMENTED | `Conversation`/`Message` aggregate, `SendChatMessageUseCase` (`POST /v1/chat`) and `GetConversationUseCase` (`GET /v1/chat/conversations/:id`). Debit-after-call settlement (no pre-call cost reservation — a documented, deliberate choice, ADR-0014), client-`messageId` idempotency guard. No model-selection-per-conversation (client cannot request a specific provider). |
| **Memory** | NOT IMPLEMENTED | No long-term memory, no embeddings, no vector store, no summarization of past conversations exists anywhere in this codebase. Not designed, not scoped. |
| **Knowledge** | NOT IMPLEMENTED | No knowledge-base/RAG concept exists in this codebase. |
| **Content** | NOT IMPLEMENTED (out of scope) | "Content" as a product concept belongs to the separate AIFA Content Studio workstream (governed by `00_System/AIFA_CONSTITUTION.md`), explicitly not part of this Platform per the D-011/D-020 separation decision. |
| **Measurement** | PARTIALLY IMPLEMENTED | Usage Tracking (`UsageEvent`, `GET /v1/usage`) records per-call tokens/cost/price/provider and exposes a paginated history read. **No analytics, dashboards, aggregation/rollup, or alerting exists** — it is a raw event log with a list endpoint, nothing more. |
| **Workflow** | NOT IMPLEMENTED | No workflow/orchestration engine, no multi-step agent pipeline exists. |
| **Agents** | NOT IMPLEMENTED | No autonomous-agent concept exists in this codebase — Chat is a single request/response exchange with one provider call per message, not an agent loop. |
| **Frontend** | NOT IMPLEMENTED (product surface) | `apps/web` has exactly three source files: a static marketing page, a static dashboard placeholder, and a server-side health-check proxy route. No auth screens, no chat UI, no wallet/usage UI, no client-side data fetching from any of the six real backend endpoints. Zero frontend tests. |
| **API** | IMPLEMENTED (for the six built features) | REST + OpenAPI (Swagger at `/v1/docs`, unauthenticated — see `08_SECURITY_REVIEW.md`). 12 real endpoints across health/auth/wallet/chat/usage. `class-validator`-backed request validation is wired globally (`ValidationPipe`). |
| **Database** | IMPLEMENTED (schema) / UNVERIFIED (runtime) | Full Prisma schema for all six bounded contexts, 5 migrations generated via `prisma migrate diff` (schema-to-schema, no live DB needed for generation). **No migration has ever been applied (`prisma migrate deploy`) against a real running Postgres** — no Postgres instance has ever been available in any environment this codebase was developed in. All persistence-layer code is verified only via in-memory test doubles and reading generated SQL, never a live query. |
| **Infrastructure** | PARTIALLY IMPLEMENTED | `infra/docker/docker-compose.yml` defines Postgres + Redis + both apps for local dev, with healthchecks. Never actually run (no Docker daemon available). No staging/production infrastructure exists or has been decided. |
| **Docker** | PARTIALLY IMPLEMENTED (unverified) | Both Dockerfiles use the documented `turbo prune --docker` pattern and have inline comments explaining two real bugs found and fixed by inspection (missing `tsconfig.base.json`, missing `prisma/schema.prisma` at install time). **Neither Dockerfile has ever actually been built** — no Docker daemon available. Neither runs as a non-root user (no `USER` directive in either Dockerfile). |
| **CI/CD** | PARTIALLY IMPLEMENTED | Workflow file exists at the correct root-level path (`.github/workflows/aifa-platform-ci.yml`), correctly scoped (`paths: Platform/**`), and provisions ephemeral Postgres+Redis services. **It has never executed even once** — the local `main` branch is 8 commits ahead of `origin/main`; nothing has ever been pushed to GitHub. No CD (continuous deployment) exists at all — no deploy step, no environment promotion, no staging/production target. |
| **Observability** | PARTIALLY IMPLEMENTED | Structured JSON logging (`pino`) exists but is **not wired into Nest's DI** — only the global exception filter receives a logger instance; successful requests are never logged with their correlation id, so request tracing is half-built. Real health/readiness checks exist (`/v1/health`, `/v1/health/ready`) covering DB, Redis, and AI providers. No tracing (no OpenTelemetry SDK wiring despite a documented strategy), no metrics export, no alerting. |
| **Security** | PARTIALLY IMPLEMENTED | See `08_SECURITY_REVIEW.md` for full detail. Real: password hashing, JWT signing/verification, refresh-token rotation/reuse-detection, global input validation, CORS allowlist, Prisma-only DB access (no raw SQL). Missing: rate limiting, security headers (no `helmet` or equivalent), secret-scanning in CI, RBAC/authorization beyond ownership, non-root Docker users, dependency-vulnerability scanning (no Dependabot/Renovate config found). |
| **Configuration** | IMPLEMENTED | `@aifa/config`'s zod schema validates every env var at boot and fails fast with a clear error listing every violation. Real, tested (`load-config.spec.ts`). |
| **Deployment** | NOT IMPLEMENTED | No staging or production environment, no orchestrator (Kubernetes/ECS/etc.) decision, no CD pipeline, no rollback strategy beyond what's described in prose in `ci-cd-pipeline.md`. |
| **Testing** | PARTIALLY IMPLEMENTED | ~104 unit/integration-via-fake tests across the monorepo, all passing. Zero true integration tests against a real database or Redis instance. Zero end-to-end tests. Zero frontend tests. No coverage tool configured, so the actual percentage of code exercised is unknown. |
